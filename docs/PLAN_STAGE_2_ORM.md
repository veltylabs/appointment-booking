## 1. repository.go

This is the **custom** wrapper file, written by you, the developer. It should import `tinywasm/orm` and use both the direct ORM engine (`db.Create()`, `db.Update()`) **AND** the typed query helpers generated in `model_orm.go` (like `ReadOne...()`, `ReadAll...()`). Follow the pattern from `business-hours/repository.go` or similar repositories.

Tables:
- `employee_service_config`
- `reservation`
- `workcalendar_config`
- `workcalendar_weekly`
- `workcalendar_exception`

Required operations per table:

### employee_service_config
- `InsertEmployeeServiceConfig(cfg EmployeeServiceConfig) error`
- `GetEmployeeServiceConfig(id string) (EmployeeServiceConfig, error)`
- `ListEmployeeServiceConfigByStaff(tenantID, staffID string) ([]EmployeeServiceConfig, error)`
- `UpdateEmployeeServiceConfig(cfg EmployeeServiceConfig) error`

### reservation
- `InsertReservation(r Reservation) error`
- `GetReservation(id string) (Reservation, error)`
- `ListReservationsByStaff(tenantID, staffID string, from, to int64) ([]Reservation, error)`
- `ListReservationsByClient(tenantID, clientID string) ([]Reservation, error)`
- `UpdateReservationStatus(id, status, updatedBy string, updatedAt int64, revision int) error`

*Note: The reservation mapping should ideally include a Unique Constraint on `(staff_id, reservation_date, reservation_time)` taking into account status, but since status is mutable, we will manage atomic locks or DB Transactions via the service during creation.*

**`revision` field lifecycle (optimistic concurrency):**
- Initial value: `0` — set by `InsertReservation`.
- Incremented by 1 on every `UpdateReservationStatus` call.
- Since `db.Update()` in `tinywasm/orm` only accepts a model pointer and updates by Primary Key (no dynamic `Where` clauses), **optimistic concurrency MUST be validated manually inside a DB Transaction**.
- Steps for `UpdateReservationStatus`:
  1. Start a transaction (`db.Tx(func(tx orm.TxExecutor) error { ... })` or equivalent).
  2. Read the current reservation by ID using the generated `ReadOne...` function.
  3. Compare the current reservation's `revision` against expected `cmd.Revision`. If they do not match, return `ErrConflict` (sentinel error defined in this package) to abort the transaction.
  4. Apply changes and increment the `revision` field by 1.
  5. Call `db.Update(tx, &model)` (using the transaction executor).
- The caller (service layer) must reload the reservation and retry with the new revision if `ErrConflict` is returned.

**String Primary Keys (PKs):**
- `tinywasm/orm` DOES NOT auto-generate string IDs.
- Any new entity with a string primary key MUST have its ID generated using `github.com/tinywasm/unixid.Generate()` BEFORE calling `db.Create()` or inserting. This logic should be implemented in the models' `Insert*` operations in `repository.go`.

**Transactions:**
- Both ORM connections must support Transactions via the driver for atomic Optimistic Concurrency checks and for complex FSM+Reschedule events.
